# Agentic BI 驱动的多表电商运营分析与决策智能系统项目报告

成员：2354283 黄迪、2351269 黄一和、2352048 夏浩博

## 摘要

本项目基于 Brazilian E-Commerce Public Dataset by Olist，构建了一个面向电商运营场景的 Agentic BI 系统。系统通过 Streamlit 提供自然语言交互界面，使用 LangGraph 编排多个智能体，连接 MySQL 中的 9 张原始业务表与 6 张预聚合视图，实现描述性分析、诊断性分析、预测性分析和规范性决策建议。系统支持月度销售趋势、州级销售分布、支付方式分析、配送表现、商品重量与运费关系、评论情感与主题建模、What-if 卖家筛选、异常检测和多轮关联分析。

在性能优化方面，项目将高频查询沉淀为预聚合视图层。以“2017 年各州销售额排名”为例，使用 `mv_state_sales` 的中位查询耗时为 30.49 ms，而基础表 `orders + order_items + customers` 实时 JOIN 的中位耗时为 363.98 ms，预聚合视图约快 11.94 倍，且两种查询结果完全一致。

## 1. 项目背景与动机

传统 BI 系统通常依赖固定报表和人工配置仪表盘。对于业务人员来说，当问题从“销售额是多少”进一步变成“为什么某些州配送慢”“哪些卖家拖累评分”“如果筛掉低评分卖家会怎样”时，传统报表往往需要数据人员重新写 SQL、制作图表、解释结果，响应链路较长。

Agentic BI 的核心思想是让大语言模型和多智能体系统主动参与业务分析流程：用户用自然语言提出问题，系统自动完成问题理解、任务拆解、SQL 查询、图表生成、预测建模、评论文本分析和决策建议生成。本项目选择 Olist 多表电商数据集，是因为它同时包含订单、商品、客户、卖家、支付、评论、地理位置等多个运营维度，能够充分体现多表 JOIN、预聚合视图、预测模型和文本洞察在电商运营分析中的价值。

本项目的目标包括：

- 支持非技术用户通过中文自然语言进行电商运营分析。
- 在 MySQL 中维护原始 9 张业务表和预聚合视图层，提高高频分析查询效率。
- 使用 LangGraph 构建多智能体协作流程，覆盖数据分析、NLP、预测、可视化、What-if、异常检测和决策建议。
- 自动生成不少于 6 类图表，并在 Web 页面中展示分析结果、原始 SQL 结果和业务建议。
- 通过性能对比证明预聚合视图层对 Agentic BI 交互式分析的加速效果。

## 2. 系统架构设计

系统整体采用“前端交互层 + Agent 编排层 + 数据查询层 + 可视化与决策层”的架构。用户在 Streamlit 前端输入自然语言问题后，Coordinator Agent 识别问题类型并生成任务计划，随后由 Data Analyst、NLP Insight、Predictor、Scenario、Visualizer 和 Decision Agent 协作完成分析。

图 1 展示了系统总体架构。系统中最关键的设计是数据层分为两部分：

- 原始 Olist 9 表：保留完整明细，用于无法被预聚合视图覆盖的细粒度问题。
- 预聚合视图层：对月度销售、州级销售、品类销售、配送表现、卖家绩效、支付方式等高频维度提前汇总，作为 Data Analyst Agent 的优先查询对象。

### 4.3 预处理步骤

数据预处理主要包括：

1. 字段类型规范化：将订单时间、支付时间、评论时间等字段转换为日期时间类型，将价格、运费、评分等字段转换为数值类型。
2. 多表拆分与入库：将 Olist 数据拆分并写入 MySQL 中的 9 张业务表，保证主外键字段可用于 JOIN。
3. 缺失值处理：配送时间为空的订单不参与准时率和配送时长计算；评论文本为空的记录不参与 NLP 文本分析。
4. 品类翻译处理：通过 `product_category_name_translation` 将葡萄牙语品类名映射为英文品类名，若无翻译则保留原始品类。
5. 业务口径过滤：预聚合视图中销售、配送和支付分析主要保留 `order_status = 'delivered'` 的已完成订单。
6. 构建预聚合视图层：根据高频分析维度生成 6 张物化汇总表，减少实时多表 JOIN 的成本。

## 5. 预聚合视图设计专节

### 5.1 视图清单与 Agent 使用方式

| 视图名 | 粒度 | 核心字段 | Agent 使用场景 |
|---|---|---|---|
| `mv_monthly_sales` | `ym` | `total_gmv`, `total_orders`, `avg_basket`, `total_freight` | 月度销售趋势、GMV、预测模型输入 |
| `mv_state_sales` | `ym + customer_state` | `total_gmv`, `total_orders`, `unique_customers` | 各州销售额排名、区域市场对比、地图展示 |
| `mv_category_sales` | `ym + product_category_english` | `total_gmv`, `total_orders`, `avg_price` | 品类销售表现、Top 品类分析 |
| `mv_delivery_perf` | `ym + customer_state` | `avg_delivery_days`, `on_time_rate`, `delayed_orders` | 准时交付率、配送延迟诊断 |
| `mv_seller_perf` | `ym + seller_id` | `seller_state`, `total_gmv`, `total_orders`, `avg_review_score` | 卖家绩效、低评分卖家定位、What-if 筛选 |
| `mv_payment_dist` | `ym + payment_type` | `total_transactions`, `avg_installments`, `total_value` | 支付方式偏好、分期模式分析 |

Data Analyst Agent 的系统提示词中明确写入“View-first strategy”：凡是用户问题匹配到视图粒度，就必须优先查询 `mv_*` 预聚合视图；只有当视图无法覆盖维度时，才回退基础表。例如商品重量与运费关系需要 `products.product_weight_g` 与 `order_items.freight_value` 的明细关系，因此回退基础表；而月度销售、州级销售、支付方式、配送准时率等问题则优先使用视图。

### 5.2 SQL 定义

#### 5.2.1 `mv_monthly_sales`

```sql
DROP TABLE IF EXISTS mv_monthly_sales;
CREATE TABLE mv_monthly_sales (
    ym VARCHAR(7) PRIMARY KEY,
    total_gmv DOUBLE,
    total_orders BIGINT,
    avg_basket DOUBLE,
    total_freight DOUBLE
);

INSERT INTO mv_monthly_sales (ym, total_gmv, total_orders, avg_basket, total_freight)
SELECT
    DATE_FORMAT(o.order_purchase_timestamp, '%Y-%m') AS ym,
    SUM(oi.price) AS total_gmv,
    COUNT(DISTINCT o.order_id) AS total_orders,
    ROUND(SUM(oi.price) / COUNT(DISTINCT o.order_id), 2) AS avg_basket,
    SUM(oi.freight_value) AS total_freight
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
WHERE o.order_status = 'delivered'
GROUP BY ym
ORDER BY ym;
```

#### 5.2.2 `mv_state_sales`

```sql
DROP TABLE IF EXISTS mv_state_sales;
CREATE TABLE mv_state_sales (
    ym VARCHAR(7),
    customer_state VARCHAR(8),
    total_gmv DOUBLE,
    total_orders BIGINT,
    unique_customers BIGINT,
    PRIMARY KEY (ym, customer_state)
);

INSERT INTO mv_state_sales (ym, customer_state, total_gmv, total_orders, unique_customers)
SELECT
    DATE_FORMAT(o.order_purchase_timestamp, '%Y-%m') AS ym,
    c.customer_state,
    SUM(oi.price) AS total_gmv,
    COUNT(DISTINCT o.order_id) AS total_orders,
    COUNT(DISTINCT c.customer_unique_id) AS unique_customers
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.order_status = 'delivered'
GROUP BY ym, c.customer_state
ORDER BY ym, c.customer_state;
```

#### 5.2.3 `mv_category_sales`

```sql
DROP TABLE IF EXISTS mv_category_sales;
CREATE TABLE mv_category_sales (
    ym VARCHAR(7),
    product_category_english VARCHAR(256),
    total_gmv DOUBLE,
    total_orders BIGINT,
    avg_price DOUBLE,
    PRIMARY KEY (ym, product_category_english)
);

INSERT INTO mv_category_sales (ym, product_category_english, total_gmv, total_orders, avg_price)
SELECT
    DATE_FORMAT(o.order_purchase_timestamp, '%Y-%m') AS ym,
    COALESCE(t.product_category_name_english, p.product_category_name) AS product_category_english,
    SUM(oi.price) AS total_gmv,
    COUNT(DISTINCT o.order_id) AS total_orders,
    ROUND(AVG(oi.price), 2) AS avg_price
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
LEFT JOIN product_category_name_translation t ON p.product_category_name = t.product_category_name
WHERE o.order_status = 'delivered'
GROUP BY ym, product_category_english
ORDER BY ym, product_category_english;
```

#### 5.2.4 `mv_delivery_perf`

```sql
DROP TABLE IF EXISTS mv_delivery_perf;
CREATE TABLE mv_delivery_perf (
    ym VARCHAR(7),
    customer_state VARCHAR(8),
    avg_delivery_days DOUBLE,
    on_time_rate DOUBLE,
    delayed_orders BIGINT,
    PRIMARY KEY (ym, customer_state)
);

INSERT INTO mv_delivery_perf (ym, customer_state, avg_delivery_days, on_time_rate, delayed_orders)
SELECT
    DATE_FORMAT(o.order_purchase_timestamp, '%Y-%m') AS ym,
    c.customer_state,
    ROUND(AVG(DATEDIFF(o.order_delivered_customer_date, o.order_purchase_timestamp)), 1) AS avg_delivery_days,
    ROUND(
        SUM(CASE WHEN o.order_delivered_customer_date <= o.order_estimated_delivery_date THEN 1 ELSE 0 END)
        / COUNT(*) * 100, 1
    ) AS on_time_rate,
    SUM(CASE WHEN o.order_delivered_customer_date > o.order_estimated_delivery_date THEN 1 ELSE 0 END) AS delayed_orders
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.order_status = 'delivered'
  AND o.order_delivered_customer_date IS NOT NULL
  AND o.order_estimated_delivery_date IS NOT NULL
GROUP BY ym, c.customer_state
ORDER BY ym, c.customer_state;
```

#### 5.2.5 `mv_seller_perf`

```sql
DROP TABLE IF EXISTS mv_seller_perf;
CREATE TABLE mv_seller_perf (
    ym VARCHAR(7),
    seller_id VARCHAR(64),
    seller_state VARCHAR(8),
    total_gmv DOUBLE,
    total_orders BIGINT,
    avg_review_score DOUBLE,
    PRIMARY KEY (ym, seller_id)
);

INSERT INTO mv_seller_perf (ym, seller_id, seller_state, total_gmv, total_orders, avg_review_score)
SELECT
    DATE_FORMAT(o.order_purchase_timestamp, '%Y-%m') AS ym,
    s.seller_id,
    s.seller_state,
    SUM(oi.price) AS total_gmv,
    COUNT(DISTINCT o.order_id) AS total_orders,
    ROUND(AVG(r.review_score), 2) AS avg_review_score
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN sellers s ON oi.seller_id = s.seller_id
LEFT JOIN order_reviews r ON o.order_id = r.order_id
WHERE o.order_status = 'delivered'
GROUP BY ym, s.seller_id, s.seller_state
ORDER BY ym, s.seller_id;
```

#### 5.2.6 `mv_payment_dist`

```sql
DROP TABLE IF EXISTS mv_payment_dist;
CREATE TABLE mv_payment_dist (
    ym VARCHAR(7),
    payment_type VARCHAR(32),
    total_transactions BIGINT,
    avg_installments DOUBLE,
    total_value DOUBLE,
    PRIMARY KEY (ym, payment_type)
);

INSERT INTO mv_payment_dist (ym, payment_type, total_transactions, avg_installments, total_value)
SELECT
    DATE_FORMAT(o.order_purchase_timestamp, '%Y-%m') AS ym,
    p.payment_type,
    COUNT(*) AS total_transactions,
    ROUND(AVG(p.payment_installments), 1) AS avg_installments,
    SUM(p.payment_value) AS total_value
FROM orders o
JOIN payments p ON o.order_id = p.order_id
WHERE o.order_status = 'delivered'
GROUP BY ym, p.payment_type
ORDER BY ym, p.payment_type;
```

### 5.3 性能对比

本项目选择同一业务问题进行性能对比：**2017 年各州销售额排名**。

| 查询方式 | SQL 来源 | 中位耗时 | 平均耗时 | 返回州数 |
|---|---|---:|---:|---:|
| 预聚合视图查询 | `mv_state_sales` | 30.49 ms | 54.96 ms | 27 |
| 基础表实时 JOIN | `orders + order_items + customers` | 363.98 ms | 371.41 ms | 27 |

<img src="assets/perf_view_vs_base_2017_state_sales.png" alt="预聚合视图性能对比" width="760">

两种查询返回结果完全一致：均返回 27 个州，GMV 最大差异为 0.0，订单数最大差异为 0。预聚合视图约快 11.94 倍，说明提前按业务粒度汇总能够显著降低 Agent 查询过程中的实时 JOIN 和 GROUP BY 成本。

预聚合视图 SQL：

```sql
SELECT
    customer_state,
    ROUND(SUM(total_gmv), 2) AS total_gmv,
    SUM(total_orders) AS total_orders
FROM mv_state_sales
WHERE ym LIKE '2017%'
GROUP BY customer_state
ORDER BY total_gmv DESC;
```

基础表 JOIN SQL：

```sql
SELECT
    c.customer_state,
    ROUND(SUM(oi.price), 2) AS total_gmv,
    COUNT(DISTINCT o.order_id) AS total_orders
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.order_status = 'delivered'
  AND o.order_purchase_timestamp >= '2017-01-01'
  AND o.order_purchase_timestamp < '2018-01-01'
GROUP BY c.customer_state
ORDER BY total_gmv DESC;
```

## 6. 智能体实现和多智能体调度方法

### 6.1 Agent 角色设计

| Agent | 主要职责 | 输入 | 输出 |
|---|---|---|---|
| Coordinator Agent | 理解自然语言问题，判断分析类型，生成任务计划 | 用户问题、上下文 | `analysis_type`, `task_plan` |
| Data Analyst Agent | 根据问题生成 SQL，优先命中预聚合视图，必要时回退基础表 | 子任务、数据字典、视图说明 | SQL、查询结果、数据摘要、查询策略 |
| NLP Insight Agent | 对评论文本做情感分析、主题提取和决策信号总结 | 评论文本、评分、品类 | 正负面比例、差评主题、关键词、行动建议 |
| Predictor Agent | 基于历史月度销售序列预测未来销售趋势 | `mv_monthly_sales` | 预测值、置信区间、趋势方向 |
| Scenario Agent | 运行 What-if 卖家筛选和异常检测 | 用户问题、数据结果、视图指标 | What-if 模拟结果、异常告警 |
| Visualizer Agent | 选择图表类型并生成图表 | 数据结果、问题类型 | 折线图、柱状图、地图、热力图、散点图、词云等 |
| Decision Agent | 融合数据、预测、NLP 和场景分析，给出运营建议 | 各 Agent 中间结果 | 业务解释、风险提示、行动方案 |
| Synthesis | 汇总最终回答 | 全部状态 | Markdown 格式最终输出 |

### 6.2 调度逻辑

系统使用 LangGraph 的 `StateGraph` 定义节点和边。各 Agent 共享 `AgentState`，其中包含用户问题、任务计划、SQL 结果、NLP 结果、预测结果、场景分析结果、图表路径、决策建议和最终回答。

调度逻辑包括：

1. 用户提交问题后，记忆模块先判断是否为追问。如果问题中出现“这些州”“刚才的卖家”“继续分析”等表达，则将上一轮的重点实体和结论注入当前问题。
2. Coordinator Agent 生成任务计划，并判断问题属于描述性、诊断性、预测性或规范性分析。
3. Data Analyst Agent 执行数据查询。若命中预聚合视图粒度，则使用 `mv_*`；否则使用基础表 JOIN。
4. 路由函数根据问题关键词决定是否触发 NLP、预测或场景分析。
5. Visualizer Agent 根据问题和结果自动选择图表。
6. Decision Agent 将数据摘要、NLP 主题、预测结果、What-if 结果和异常告警整合为业务建议。
7. Synthesis 输出最终回答，Streamlit 将回答、图表、加分项结果和原始数据分别展示在不同页签。

### 6.3 加分功能实现

评论文本情感分析与主题建模：NLP Insight Agent 从 `order_reviews` 读取评论文本，结合评分划分正负面评论，计算平均评分、正负面比例，并通过关键词规则提取物流延迟、商品质量、错发漏发、客服退款、价格感知等主题，最终将主题转化为 Decision Agent 可使用的决策信号。

What-if 分析：Scenario Agent 检测到 “what-if / 如果 / 移除 / 筛选卖家” 等问题后，调用 `what_if_remove_worst_sellers`，基于 `mv_seller_perf` 找出低评分卖家，估算移除后平台平均评分变化和 GMV 风险。

异常检测 Agent：异常检测模块对月度 GMV、州级销售额下降和品类评分下降进行扫描，使用 z-score、历史均值对比和评分下降阈值生成告警，并给出 action hint。

记忆模块：系统通过 MemorySaver 和自定义 thread memory 保存上一轮问题、重点州、重点卖家、场景分析结果和建议摘要，使后续追问能够关联上一轮分析对象。

## 8. 技术挑战与解决方案

| 技术挑战 | 表现 | 解决方案 |
|---|---|---|
| 多表 JOIN 查询成本高 | 订单、商品、客户、支付、评论等表频繁 JOIN，交互式分析容易变慢 | 设计 6 张预聚合视图，将高频分析维度提前物化 |
| LLM 生成 SQL 容易选错表或字段 | 例如把 `customer_state` 写到 `orders` 表，或使用 PostgreSQL 语法 | 在 Data Analyst Prompt 中加入字段归属表、MySQL 语法规则和 view-first checklist |
| 预聚合视图与基础表结果需一致 | 性能优化不能牺牲结果准确性 | 使用同一问题对比视图 SQL 和基础表 JOIN SQL，校验州数、GMV 和订单数差异为 0 |
| 中文自然语言和英文数据字段之间存在语义映射 | 用户输入中文，数据库字段和品类多为英文或葡萄牙语 | Coordinator 识别中文意图，Data Analyst 使用数据字典和视图说明映射到 SQL |
| 评论文本为葡萄牙语 | 直接中文关键词无法分析评论 | 对评论文本进行规范化、葡萄牙语关键词提取，并结合 LLM 生成主题摘要 |
| What-if 结果不能由 LLM 编造 | 业务场景需要可解释和可复现的数值 | Scenario Agent 直接从 `mv_seller_perf` 和明细数据计算评分变化与 GMV 风险 |
| 多轮追问容易污染普通问题 | 如果无条件注入历史上下文，会导致不相关问题误触发 Agent | 只在判断为 follow-up 问题时使用 memory context，普通问题保持独立 |
| 前端结果较多，页面容易拥挤 | 图表、原始数据、加分项和分析文本同时展示 | 使用 Charts、Bonus Agents、Analysis、Raw Data 页签分区展示，并为右侧结果区设置滚动容器 |

## 10. 总结

本项目完成了一个可运行的 Agentic BI 电商运营分析系统。系统能够通过自然语言完成多表数据查询、预聚合视图加速、自动可视化、销售预测、评论情感分析、What-if 模拟、异常检测和多轮关联分析。与传统静态 BI 报表相比，本系统更强调“问题驱动”的分析过程：用户无需直接编写 SQL，系统即可根据问题自动选择视图、查询数据、生成图表并给出可执行建议。

从实验结果看，预聚合视图层显著提升了查询效率，在“2017 年各州销售额排名”任务中获得约 11.94 倍加速，同时保持结果一致。该设计证明，在 Agentic BI 场景下，将 LLM Agent 与稳定的数据建模和预聚合视图结合，可以同时兼顾自然语言交互的灵活性和数据分析结果的可靠性。
